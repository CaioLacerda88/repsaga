0 · What changed since Round 1

Round 1 locked the aesthetic (Concept B). Round 2 resolved structural questions that surfaced when both agents (PO + UX-critic) audited Concept B against every realistic session state. Six decisions were made; each is adjudicated below.

Question Decision Rationale
Is Beat 1 always identical? No — 4 tier-aware micro-variants of one structural beat. Both agents pushed back on "always same." Same structure (XP slam, diagonal slash, hold), but copy + tension budget vary: day-zero · baseline · PR-anticipatory · class-change-anticipatory. Max-combo gets a 120ms anticipation pre-roll. No additional beats, just one parameter.
Beat 2 multi-stat choreography (≥2 body parts) Hybrid: ≤2 BPs = sequential hue floods, ≥3 BPs = cascade grid (dominant hue + 3 cascade chips). Pure sequential cuts (UX-critic preference) blow timing budget at 4+ BPs (~6s of Beat 2 alone). PO's cascade grid keeps the cut to ~2s regardless of N. Threshold at 3 BPs is the breakpoint between "I had a focused session" and "I worked everything." Both modes feel narratively correct.
Beat 3 reward stacking when multiple events fire Max 2 Beat 3 cuts. Rank-up promoted into Beat 2 elevated variant. Level-up folded into Beat 1 copy. Class change always closes. PO's adjudication: Beat 3 events do NOT stack cinematically — they chain. PR + class change is the only legitimate two-cut combo. Rank-up + Beat 2 fusion is more narratively coherent than a separate cut. Level-up overlay already fires mid-workout, so Beat 1 copy ("NÍVEL 23. A SAGA CONTINUA.") absorbs the announcement.
Share card direction Killed Round 1's "Rank Badge Drop." User shoots a real gym photo; RepSaga composites the saga overlay on top. Strava-style, native render. Round 1's generated card felt like a Hevy social tile (informative but emotionally flat). The gym mirror selfie + plate shot + pump shot ARE the BR WhatsApp Status vernacular. The overlay is the differentiator, not the source image.
Discreet mode (user declines photo) First-class fallback — not a degraded substitute. Concept-B abyss + dominant-hue flood, saga data composed in same grammar as the cinematic cuts. Users who never share photos still get an on-brand shareable. The discreet card IS a film still. Same diagonal slash, same hero numeric, same hue identity — minus the photo zone.
Mid-workout celebration overlays (4 existing widgets) Redesign all four as 400ms thin Concept B hue-flashes (no card chrome, no glow, no border-radius, no interactive CTA, auto-dismiss). Title EQUIP CTA migrates OUT of mid-workout to the post-session summary panel. Existing class_change_overlay.dart (1600ms sigil ceremony) → shipped as PR 29.5 before 30a. UX-critic call after both options analyzed. Gym workout = dungeon run with mid-run drops + post-run tally, not single boss fight. First-awakening MUST stay mid-workout for causal-link tutorial value. Class-change withheld for post-session = "paperwork delay," not climax (user has been acting as new class for last 10 min). Variable-ratio reinforcement breaks if all rewards saved for fixed end-screen. Concept-B cinematic at session end + 400ms environmental signals mid-workout = complementary, not redundant.
Anti-AI render rules locked. The UX-critic flagged the most dangerous moment of this round as the HTML render itself. Guard-rails: clip-path: polygon() for diagonal slashes (not box-shadow or border-radius). rgba(13,3,25,0.95) flat fills for data zones (not backdrop-filter: blur). Zero border-radius on data strips. No gradient mesh, no glassmorphism, no soft pastel achievement toasts. Confetti, particles, "level up!" burst badges — all banned.

Why cascade vs sequential at 3+ BPs?

  • Sequential at 4 BPs = ~3.0s of cuts. Combined with Beat 1 + Beat 3 → 14s baseline. Violates "baseline must be tight."
  • Cascade reads as one declarative scene with the dominant BP as hero, not four equally weighted ceremonies.
  • Honors the "every BP gets surfaced" invariant without inflating motion budget.
  • If 5–6 BPs trained, cascade still works — top 1 hero + cascade list up to 4 rows (truncation at 4 with "+N mais" pill).

Dominant BP selection

Highest XP earned this session. If tied → highest current rank. If still tied → arbitrary stable order (alphabetical pt-BR).

Why a deeper post-session cut if mid-workout already flashed?

  • Mid-workout flash = environmental signal ("this just happened, your class is now Bulwark, back to your set"). 400ms, no interaction, auto-clears.
  • Post-session cut = ceremony ("here's what Bulwark means: you are the chest specialist; your flavor is O peito é o escudo"). 1.5s hold, the moment lingers.
  • Same event, two attentional layers. The flash anchors causality; the cinematic does identity-reveal. Neither is redundant.
  • The bottom italic flavor line is the post-session-only differentiator — mid-workout never has time for prose.

Why Rajdhani 42 (not 26-stacked)?

Earlier draft stacked "BULWARK / DESPERTOU." on two lines at 26px each. The single-line "BULWARK" at 42px + "DESPERTOU." subline below at 13px gives the class name unambiguous primacy — the class is the identity, "desperto" is the verb. Class chip on the summary panel afterward reads the same class name with the same weight, so the eye carries identity continuity.

4½ · Mid-workout flash layer — RETIRED (Path A pivot, 2026-05-22)

Path A pivot (2026-05-22). The original §4½ specced a "dual-loop" model: mid-workout flashes during the workout for variable-ratio reinforcement, post-session ceremony at finish for the deep reveal — same event, two attentional contexts, claimed to be complementary not redundant. On-device verification (Galaxy S25 Ultra, PR 29.5 debug APK) surfaced the gap: the architecture only supports session-finish event emission (events are computed from a pre-vs-post session diff inside record_session_xp_batch, not per-set). So in practice the "mid-workout" flashes fired ~200ms before the post-session screen mounted — same moment, same attention context, low-res preview of the high-res cinematic that fires 4 seconds later. The dual-loop thesis only holds with true per-set firing; without it, the flash layer is redundant pre-roll. Decision: kill the mid-workout flash layer entirely. Phase 30's post-session cinematic ceremony (Beats 1–5, see §5–§8) carries the full celebration for ALL events. The legacy mid-workout overlay widgets (class_change_overlay, level_up_overlay, first_awakening_overlay, title_unlock_sheet, rank_up_overlay) still retire in PR 29.5 (they contradicted Concept B grammar); they're replaced by NOTHING — the events skip the flash layer entirely and pass through to the post-session screen.

What survives from the original §4½ spec

  • CelebrationEvent.personalRecord variant — kept on the sealed union. Consumed by post-session Beat 4 (PR cards), not by a mid-workout flash.
  • CelebrationQueue.SlotPolicy enum — kept. Drives post-session event ordering / coalescing / serialization.
  • Retirement of 5 legacy mid-workout overlay widgets — kept. Concept B grammar incompatibility was real regardless of the dual-loop question.
  • ARB key additions — most retired (no eyebrow / hero strings needed for a flash layer that doesn't exist); a few survive if Beat 2/3/4 reuses them.

What got cut in the pivot

  • thin_flash_overlay.dart — deleted; no replacement widget. The post-session screen carries the entire celebration in §5–§8 cuts.
  • scripts/check_thin_flash_no_glow.sh CI gate — deleted; no thin-flash widget to gate.
  • Mid-workout playback inside celebration_orchestrator.dart — simplified to a pass-through (or fully retired). The post-session screen consumes CelebrationQueueResult directly.
  • Thin-flash semantics selectors — removed from test/e2e/helpers/selectors.ts; no DOM surface to select.

If we ever revive mid-workout flashes (future scope, not Phase 30)

Path B (hybrid) and Path C (full mockup spec) were considered and rejected for Phase 30. They require per-set XP delta computation + per-set state diff + emission inside recordSet() — a 3–7 day architecture expansion sensitive to the same NUMERIC-rounding parity bugs Phase 29 just spent three PRs untangling. If post-launch data shows users wanting more frequent mid-workout reinforcement (variable-ratio gym gamification IS the differentiator vs Strong/Hevy/Fitbod), revisit then — the PersonalRecordEvent variant + SlotPolicy enum kept in PR 29.5 are the scaffolding that would be needed.

Title equip CTA — moved out of mid-workout entirely

The existing title_unlock_sheet.dart has an "EQUIP" CTA that demands user action mid-workout. This was always the weakest mid-workout overlay (it interrupted set logging with a decision the user wasn't ready to make). Phase 30 strips this: mid-workout title shows the flash only (no EQUIP). The EQUIP affordance migrates to the post-session summary panel, where the user has time to consider it. New title earned → summary panel shows a "Novo título · PILAR DE FERRO · [EQUIPAR]" row. User can equip there OR dismiss and visit the titles screen later. Mid-workout never demands an action — that's the rule.

Implementation surface · PR 29.5 (before 30a)

The overlay redesign ships as PR 29.5, BEFORE 30a (post-session screen). Reason: the post-session storyboards assume the mid-workout overlays are already the new thin flashes (Beat 1 level-up copy says "confirms what already happened" — that copy only makes sense if the flash is the new thin one, not the old soft-glow card). PR 29.5 is the foundation that lets the post-session cinematic land coherently.

Implementation notes

  • Render pipeline: Flutter RepaintBoundary + RenderRepaintBoundary.toImage(pixelRatio: 3.0), JPEG encode at quality 88. Produces ~900KB–1.1MB for typical gym photo at 1080×1920.
  • Camera: image_picker for both take-photo + gallery-pick. No in-app camera UI.

  • Toggle interactions on preview: tap-to-hide XP number / tap-to-hide PR line. Lets users control disclosure (some users won't want their numbers public). Drag-to-reframe gesture on the photo zone.

8 · Implementation gaps for tech-lead

Surfaces flagged by PO + UX-critic agents during their audits. Must be addressed before Phase 30 implementation can start.

Gap · structural
CelebrationEvent has no PR variant

The CelebrationEvent sealed union in lib/features/rpg/models/celebration_event.dart currently handles rankUp, levelUp, classChange, titleUnlock, firstAwakening — but no personalRecord variant exists. celebration_event_builder.dart has no path to detect PRs because it compares rank snapshots and levels but never consults peak_loads_repository.dart.

Recommended fix: add CelebrationEvent.personalRecord(exerciseName, weight, reps, repBand, priorBest?) variant. Wire celebration_event_builder.dart to detect PRs from the peak-loads delta during workout finish. This keeps the event model as the single source of truth and lets the CelebrationQueue factor PR events into its cap logic.

Alternative (rejected): compute PR data separately in the post-session screen's own provider and inject alongside the queue. Adds a second source of truth — harder to keep mid-workout overlays and post-session screen in sync.

Gap · routing · PR 30a
Full-screen route push after workout finish

The post-session screen MUST be a full-screen route pushed onto the navigator (not a bottom sheet, not a dialog) so the cinematic can use full-bleed cuts. The existing finish_workout_coordinator.dart currently routes to home after persistence — needs to route to the new post-session screen, which then routes to home after the user taps CONTINUAR or auto-advances.

Empty-session guard (State 11) lives in this coordinator — BEFORE the post-session route push. If zero sets logged, show the disambig sheet inline; never push the post-session route.

Sequencing with PR 29.5: the coordinator drains the CelebrationQueue via celebration_player BEFORE pushing the post-session route. By the time the cinematic starts, every mid-workout thin flash has already fired and dismissed.

PR 29.5 · MUST SHIP BEFORE 30a
Mid-workout overlay redesign — 4 widgets → 1 thin-flash widget

The four legacy mid-workout overlays contradict Concept B's hard-cut grammar (centered sigil cards, soft glows, color tweens, modal backdrops, interactive CTAs). PR 29.5 retires all four and ships a single new ThinFlashOverlay widget that consumes a CelebrationEvent and renders the appropriate 400ms hue-flash variant. See §4½ for the five variants.

Gap · animation orchestration
Single timeline, not per-widget AnimationControllers

Gap · responsive text sizing

The mockup uses narrow 160px scaled cuts where text sizes have been reduced visually so the canonical sample text (BULWARK, DESBRAVADOR, 95kg × 5, +1042) fits cleanly. The real Flutter implementation must use FittedBox(fit: BoxFit.scaleDown) on every hero-tier text element so longer pt-BR strings (DESBRAVADOR 11 chars, ASCENDENTE 10 chars, PATHFINDER, multi-digit weights like 140kg × 12) don't overflow even on the smallest target viewport (320dp width). Existing precedent: class_change_overlay.dart's _ClassHeadline already uses FittedBox + per-glyph reveal (see lines 545-569 of the legacy file before retirement); apply the same pattern to the new thin_flash_overlay.dart hero text + every post-session cinematic cut hero.

  • B2 BP-hue XP delta (+618 XP) — Rajdhani 700, target ~52sp
  • B2 elevated rank slam (19) — Rajdhani 700, target ~72sp

  • B3 title name (DESBRAVADOR DO FERRO) — Rajdhani 700, target ~26sp
  • Mid-workout thin-flash hero text — Rajdhani 700, target ~32sp

Min/max font ranges: set minFontSize via FittedBox's natural behavior; do not let any hero text drop below 70% of target size (golden tests should pin this — a hero text scaled to 12sp on a 320dp viewport indicates the layout itself needs reconsideration, not just smaller type).
