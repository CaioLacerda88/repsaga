1 · The diagnosis

Today's post-session experience is a save-and-leave moment. The user finishes the session, the data persists, the screen closes. No RPG product would close a battle that way.

Blue ocean verdict from the competitor scan

The PO agent's audit of 7 fitness apps (Strong, Hevy, Fitbod, Strava, Zwift, Peloton, Apple Fitness+) found zero of them deliver an RPG-style post-session reveal:

  • Strong — static table with PR badge inline. Zero animation.
  • Hevy — "you lifted a truck!" volume copy is the only delight moment.
  • Fitbod — muscle heat map (analytically rich, emotionally flat).
  • Strava — full-screen animation post-save, but undifferentiated (same fanfare for 3 PRs or a mediocre run).
  • Zwift — career XP bar + bike unlock progress + level-up animation. Closest existing analog to what RepSaga wants, but cycling-only and visually dated.
  • Peloton — milestone-modal pattern (100th/500th ride) is psychologically correct but too rare to be a regular motivator.
  • Apple Fitness+ — best-in-space ring-close moment (haptic + visual + sound trinity), but requires Apple Watch hardware.
Verdict: RepSaga can ship the first strength-training app with an RPG-style post-session reveal. Per-body-part XP with hue accents + dual-axis progression (body-part rank + character level) + tier-aware celebration is a defensible differentiator at launch.

Why this matters for retention (research-backed)

  • +65% retention bump from achievement notifications that name identity ("you are now Iron II in Back") — fitness-app UX research 2024.
  • Endowed Progress Effect (Nunes & Drèze, JCR 2006) — partially-filled XP bar with a named next milestone dramatically increases the likelihood of return.
  • Hooked Investment Loop (Eyal) — the post-session screen is NOT the end of a session; it's the start of the next session's hook cycle. The last element the user sees must be a forward-looking gap.
  • Variable Reward Calibration (Skinner) — Habitica's failure: everything earns XP → nothing feels earned. Celebrations must tier by significance.

2 · Seven design principles (non-negotiable before any concept)

Both agents independently landed on the same core principles. These constrain every concept below — no concept can violate these and still ship.

01
The screen is the Victory Fanfare, not the receipt
Every session has a mandatory opening beat (0.8–1.2s) — flash or breath — before any data appears. The celebration is the product, not a gate to clear.
02
Tier the celebration to the achievement
Three distinct animation tiers in the state machine: Baseline (XP only), Notable (PR), Exceptional (rank-up + level-up). Habitica's trap: everything maxed → nothing feels earned.
03
Body-part hues are activation signals, not decoration
Chest pink, back sky, legs green appear ONLY when that body part was trained. Untrained body parts are absent, not grayed — presence = proof of work.
04
Plant the forward hook before dismissal
Last visible element MUST be a forward-looking gap: "32 XP para rank 19 em Costas". The Hooked investment loop + Endowed Progress Effect, deployed.
05
System voice, not hype-coach
RepSaga's system is mythic, fair, cold. It confirms, doesn't cheer. "Peito · rank 19" beats "Mandou bem, guerreiro!". Solo Leveling notification register, in pt-BR. Max 6 words per line. Exclamation marks reserved for rank-up + level-up only.
06
Skip costs a gesture, not a casual touch
Tap-to-advance-phase, not tap-to-close. Persistent "CONTINUAR" button visible from frame 0. Hold-to-skip-all for power users. FFXVI's failure (non-dismissible counters) is the anti-pattern.
07
Body-part vitality chart is trophy, not punishment
A small body silhouette showing which parts were worked (in respective hues) reframes muscle fatigue as evidence. Closes the screen before the forward hook.

Reward scarcity is structural (CI-enforced, can't violate)

heroGold (#FFB800) is rationed by scripts/check_reward_accent.sh. It appears ONLY on PRs, level-ups, and streak milestones. Every other reward (per-body-part XP, tonnage, sets) renders in body-part hues or textCream. A screen that paints everything gold trains the user to feel nothing.

REPSAGA

The feel

The session is not a battle. It is a page being written in the user's saga. Handcrafted, organic, reflective. The post-session screen presents the workout as an entry in a chronicle — typography sits in margins, lore text frames the data. PRs feel like signature moments in a manuscript, not stats.

Hierarchy of reveals

Beat 1: abyss bg fades with subtle texture overlay (400ms). Chapter header writes in letter-by-letter (8ms per character, variable speed). Beat 2: Lore-text block fades up from 8px below, 300ms ease-out per line, staggered 100ms. Beat 3: Body-part rows: ◆ dot inks in (scale 0 → 1, 200ms spring), then label + XP fade left-to-right. 150ms between rows. Beat 4: Rule line draws left to right. Total XP arrives already-inscribed (no counter — this screen doesn't count, it confirms). Beat 5: If PR, a heroGold ◆ line "inks" in. Beat 6: CONTINUAR A SAGA fades in.

Copy register · stoic narrator

  • "No quadragésimo sétimo treino, o Bulwark forçou além do limite." (lore, PR session — references actual character class)
  • "Mais um capítulo escrito. O progresso não vem de momentos — vem de aparições." (lore, no-PR)
  • "Recorde inscrito: Supino 95kg × 5."
  • "Peito ascendeu ao rank 19." (rank-up — narrated as story event)
  • "Nova era: Nível 23. A saga se aprofunda." (character level-up)
  • "De Initiate, Bulwark emerge. A classe se revela." (class change)
  • "Capítulo 1. Toda saga começa com um primeiro passo." (day-zero)

Defining moment

The chapter header writing in letter by letter — "Capítulo 47" — before any numbers appear. The session becomes a document.

Implementation risk · Medium

Letter-by-letter typing animation (custom widget). Optional texture overlay via ShaderMask or DecorationImage with BlendMode.overlay. The lore-text copy system needs ~30-50 pre-written variants keyed by session shape (PR / no-PR / level-up / streak-day / day-zero / maintenance).

User served best

Reflective lifter / story-curious / intrinsic-motivated. The user who treats training as personal narrative. Most distinctive but least gamified.

Risk to monitor

The lore-text copy system is a content-debt liability — 30-50 hand-written pt-BR variants requires copywriting investment. The painterly aesthetic conflicts slightly with RepSaga's current Arcane Ascent mechanical aesthetic. Best if RepSaga later introduces a "lore skin" toggle; risky as the only default for v1.

5 · Recommendation — Concept B

Both agents independently picked Concept B. The UX-critic argued it's the only concept that makes RepSaga feel like a product that knows what it is. The PO argued it's the only one with native WhatsApp-Status virality (the rank-badge drop is unmistakable in a stranger's feed: "what app is that?").

Why Concept B wins

  • The 33ms white flash + heroGold flood at PR cuts is unreplicable in generic fitness apps. Strava's undifferentiated fanfare is the cautionary tale (same animation for any activity — every Strava share looks the same). RepSaga's PR cut would be instantly recognizable in a WhatsApp Status feed.
  • Brazilian gym culture is hype-positive + social-media native. 36% of Brazilians watch anime; CCXP São Paulo is the world's largest geek convention. The kinetic Persona-style aesthetic translates 1:1 to this audience.
  • The cinematic cuts CAN be tier-aware. A baseline session gets a TIGHTER cut (XP + summary only, ~6s). A PR session adds the gold-flood cut (+1.5s). A rank-up gets the screen-wipe transition (+2s). A level-up adds the full hero card (+3s). The variable-reward principle is built into the choreography, not retrofitted.
  • Implementation risk is bounded. The "hardest" part is the orchestrated AnimationController chain, which is one timeline (not per-widget logic). No custom renderer. The 33ms white flash is two stacked widgets with a 1-frame visibility toggle.

Concept A as fallback

If Concept B's motion budget tests as too aggressive on mid-range Android devices (Snapdragon 6-series Brazilian-market phones), Concept A is the safe fallback. It serves the data-nerd user well, ships faster, and has lower performance risk. Concept A's share-card is less distinctive but still informative — it's a recognizable "post-game results" pattern.

Concept C — viable as a v1.1 skin, not the v1 default

The painterly journal aesthetic is genuinely beautiful and would attract a specific user segment. But it's a different brand DNA from the current Arcane Ascent mechanical aesthetic, and the lore-copy debt (30-50 hand-written pt-BR variants by session shape) is a content liability. Best as a future toggle: "Saga Chronicle" theme that the user opts into.

Reward hierarchy LOCKED (per FF/Persona/Clair Obscur reference research)

The post-session state machine fires reveals in this priority order. When multiple events coincide in one session, the higher-priority moment plays first; lower-priority moments stack below as additive beats.

  1. Class change (rarest, biggest) — already has production class_change_overlay.dart. Post-session screen integrates this; doesn't duplicate. Example: Initiate → Bulwark when chest rank crosses 5 with chest dominance.
  2. Character level-up — derived from sum of body-part ranks. Full-screen cinematic moment per Concept B/screen-wipe per Concept A.
  3. Title earned — pulls from the 90-title catalog. Rare unlocks deserve a heroGold-flooded reveal but shorter than level-up.
  4. Body-part rank-up — frequent at low ranks (every 1-2 sessions for beginners), rarer at high ranks. Concept B: body-part-hue flood + numeric rank slam ("PEITO · RANK 19"). Multiple rank-ups in one session stack as sequential cuts.
  5. PR (personal record) — per-lift. White flash → heroGold flood in Concept B. Multiple PRs in one session each get their own cut.
  6. Streak milestone — passive notifier in summary footer, no dedicated cut.
  7. Baseline XP tally — every session. Per-body-part XP reveal + character XP bar tick. Per your Q5 answer: this IS the minimum screen for zero-special-moments sessions — XP bars fill, ranks tick if they cross threshold, that's the show.

Share-card lock

If you pick Concept B, the share-card is "The Rank Badge Drop" — 9:16 aspect (WhatsApp Status native), body-part-hue dominant color field (~40% of card area), numeric rank in massive Rajdhani celebrationSize, body-part label + character class name as eyebrow, RepSaga wordmark bottom corner. Implementation: RepaintBoundary.toImage() on a purpose-built offscreen widget at 1080×1920, share via share_plus package, native Android share sheet handles WhatsApp / Instagram Stories / save-to-gallery. No camera permission, no gallery permission. v1.1 enhancement: optional user-photo backdrop with stats overlay.

6 · Eleven open questions to lock before implementation

Q1.Full-screen vs bottom-sheet?
All 3 concepts assumed full-screen takeover. Does the router architecture allow a full-screen route push after session save? Bottom-sheet would be lower-stakes but cinematically diluted.
Q2.Sound design scope
All reference games use audio as co-equal (the Victory Fanfare is as important as the screen). RepSaga has no audio layer today. Does the post-session screen justify introducing sound — even a single haptic + short chime for the rank-up moment — or does it land on visuals + haptics alone?
Q3.PR celebration ceiling for beginners
Beginners set PRs every session for 3-6 months. Should the maximal celebration be gated by a minimum (e.g., "PR > 5% over previous record") to preserve the moment? Or is maximal celebration for beginners a feature (newbie gains energy) rather than a bug?
Q4.Body-part silhouette asset for the vitality chart
Principle 7 recommends a body silhouette showing which parts were worked, in their respective hues. Do we have an existing icon set we can repurpose, or is this a new art commission for Phase 30?
Q5.Zero-special-moments session — what's the minimum screen?
User completes a maintenance session: same weights as last time, no PR, no rank-up, no level-up. What does the screen show? Does Concept B's tight baseline cut (6s, XP only + summary) feel earned, or does it feel like a participation trophy?
Q6.Reward hierarchy ordering
When multiple events fire in one session, what's the order? Proposed: character level-up > rank-up > PR > title earned > streak milestone > baseline XP tally. Lock this so the state machine is deterministic.
Q7.Rank/class taxonomy — RESOLVED 2026-05-20
Fixed. Replaced agents' fictional "Ferro/Bronze" tier metaphor with RepSaga's actual taxonomy: numeric body-part ranks (1-99) + 8 character classes (initiate / berserker / bulwark / sentinel / pathfinder / atlas / +2) derived from rank distribution + 90-title catalog for earned achievements. Class-change already has a dedicated production overlay (class_change_overlay.dart) that the post-session screen integrates rather than duplicates.
Q8.Share-card trigger placement
Does "compartilhar" appear as a persistent CTA during the post-session screen, or only on the final summary frame? If a user wants to share a specific PR moment (the PR frame, not the summary), can they tap share mid-sequence?
Q9.Rank badge asset for Concept B's share card
"The Rank Badge Drop" needs a geometric rank badge per tier. Do these assets exist in the design system, or is this a new icon commission for the UX-critic to spec?
Q10.Class name display on the share card
Share-card hero shows numeric rank ("RANK 19") + body-part name ("PEITO") + class ("BULWARK"). Numeric ranks are universally legible; class names ("Bulwark", "Sentinel") are RepSaga-specific. Should the class name appear prominently as a curiosity hook ("what's a Bulwark?") or stay subordinated? Mystery = acquisition trigger.
Q11.WhatsApp share aspect — 9:16 only, or also 1:1 for DMs?
WhatsApp Status accepts full-bleed 9:16; direct messages compress. Should the card offer two export sizes, or is 9:16 sufficient for v1?
