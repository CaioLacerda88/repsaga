Fill Remaining Gating — UX Proposals v1

Three-column comparative: today (left) + user-proposed sequential gate (middle) + alternative first-complete trigger (right). One lead recommendation. All options preserve the Fill Remaining core behavior and the existing set-row UX contract.

1. Current Behavior Analysis

The Trigger Logic (confirmed from source)

In exercise_card.dart, _hasFillableSets computes:

  1. Find the highest setNumber among all completed sets (lastCompletedNumber).
  2. If lastCompletedNumber == 0: return false (Fill Remaining hidden).
  3. Otherwise: return true if ANY set exists with !isCompleted && setNumber > lastCompletedNumber.

Why checking only the last set hides Fill Remaining: If sets are [1, 2, 3] and only set 3 is checked, lastCompletedNumber = 3. No set has setNumber > 3, so the condition returns false. Fill Remaining never appears. The user must check set 1 or set 2 first to surface the button.

The User-Reported UX Break

The user checks set 3 (the last) without first checking sets 1 or 2. Their expectation: Fill Remaining should appear to let them quickly complete sets 1 and 2 using the last completed values. The trigger logic prevents this because it assumes sequential completion. The assumption is technically correct for the original use case (user completed set 1, wants to fill 2 and 3) but breaks the reverse case (user is logging out-of-order).

Valid User Flows to Protect

  • Restarting mid-session: User did 3 sets on paper, came to log them. They tap set 3 first by habit (it was the last real set). They expect to fill back 1 and 2 from that value.
  • Failed set 1, completed 2 and 3: User marks set 2 and 3 done but skips set 1 (they failed or want to record it separately). Fill Remaining on sets below a completion is a legitimate workflow.
  • Bodyweight sets: User doing 3x10 pull-ups. Reps differ per set; they tick the higher-effort sets first. Sequential check assumption breaks for effort-based logging.

Evaluation of User Proposal (Sequential Gate)

The proposal: Grey out sets 2 and 3 until set 1 is checked. Fill Remaining behavior unchanged.

What it gets right: It prevents the confusing case where Fill Remaining disappears. It enforces a predictable logging contract. It reduces accidental taps on the wrong set row (gym-floor fat-finger risk reduced). It mirrors how most fitness apps work (Hevy, Strong enforce set-by-set completion).

What it breaks: The mid-session restart flow. The failed-set-1 flow. Any out-of-order logging. Users who prefer to log their best set first. The gate is paternalistic for experienced users who know what they are doing.

Verdict on user proposal: Acceptable as a starting point, but needs an escape valve. The grey-out should show a tap affordance on greyed sets with a tooltip (e.g., “Check set 1 first”) rather than being completely dead. Completely inert tap targets frustrate users who deliberately want to log out-of-order.

Alternative: First-Completed-Set Trigger

The proposal: All checkboxes remain active (no grey-out). Fill Remaining triggers as soon as ANY set is completed, filling all REMAINING uncompleted sets (both above AND below the completed one) using that set values. The button label could read “Fill all remaining sets.”

What it gets right: No broken flows. Full freedom of logging order. Fill Remaining becomes truly useful regardless of which set triggers it. Preserves the existing gesture contract (no UI change for most users).

What it changes: The semantics of fillRemainingSets expand from “fill sets above the last completed” to “fill all uncompleted sets.” This is a notifier behavior change, not just UI. Test coverage for the notifier must update. The button label must clarify the new scope to avoid confusion.

Verdict on alternative: More correct semantically. Lower implementation complexity than the sequential gate (no UI state machine needed). The behavior change is a one-line notifier modification. This is the preferred technical path IF the user is comfortable with the expanded Fill Remaining scope.

Reference Apps (Hevy, Strong)

  • Hevy: Enforces sequential set completion. Tap-to-complete is disabled on set N until set N-1 is done. No Fill Remaining concept; they use “Quick log” which bulk-copies all sets from the previous session.
  • Strong: All set rows are always tappable regardless of order. No Fill Remaining concept; each set row shows a check mark inline. Out-of-order logging is supported natively.
  • RepSaga position: Our Fill Remaining is a differentiator from both. The sequential-gate (Option B) moves us toward Hevy. The first-completed-set trigger (Option C) keeps us closer to Strong but with a smarter affordance.

Lead Recommendation: Option C — First-Complete Trigger

Pick Option C. It resolves the user-reported UX break without adding a new UI state machine or breaking any valid logging flow. The implementation change is a two-part modification: (1) update _hasFillableSets in exercise_card.dart to return true if any set is completed and any other set is incomplete, removing the directional constraint; (2) update fillRemainingSets in active_workout_notifier.dart to fill ALL uncompleted sets using the most recent completed set values, not just sets above the last completed one. The button label should add the count: “Preencher restantes (N series)” so the user knows exactly what will be filled before tapping.

Option B (sequential gate) is the right choice only if gym-floor accidental-tap data shows that out-of-order set completion is causing systematic mislogging. We have no such data pre-launch. The sequential gate also breaks legitimate workflows (the mid-session restart case the user themselves described). Option B should be held as a follow-up if Option C produces measurable confusion in user sessions.

On the user proposal specifically: The instinct is correct — make the trigger predictable. But the proposed solution (grey-out gate) achieves predictability at the cost of valid flows. Option C achieves the same predictability (Fill Remaining always appears after the first checked set) without the gate, simply by expanding the button scope to include all uncompleted sets regardless of position.

3. Implementation Notes for Tech-Lead (Option C)

Design notes only — not implementation specs.
  • _hasFillableSets (exercise_card.dart): Replace the current logic with: return true if sets.any((s) => s.isCompleted) AND sets.any((s) => !s.isCompleted). The directional constraint (setNumber > lastCompletedNumber) is removed. The button appears as long as at least one set is done and at least one is pending.
  • fillRemainingSets (active_workout_notifier.dart): Change from “fill sets with setNumber > lastCompletedNumber” to “fill all sets where !isCompleted” using the most recent completed set weight + reps values. This is a one-line change to the set filter in the notifier method.
  • Button label: Update from l10n.fillRemaining (static string) to a parameterized version: l10n.fillRemainingSets(count) where count = sets.where((s) => !s.isCompleted).length. This tells the user exactly how many sets will be filled before they tap. Add the ARB key fillRemainingSets (parameterized) alongside the existing fillRemaining key.
  • Test coverage: Update existing fillRemainingSets notifier tests to cover the expanded contract: (a) completing only the last set and triggering fill for sets 1..N-1, (b) completing only the first set and filling 2..N (existing behavior, now verified), (c) completing a middle set and filling all others. Widget test for _hasFillableSets must cover the out-of-order check case.
  • Semantics: The fillRemainingSetsSemantics ARB key currently reads “Fill remaining sets with last completed values.” Update to “Fill all uncompleted sets with last completed values” to reflect the expanded behavior. This is an accessibility contract change and must ship alongside the code change.
